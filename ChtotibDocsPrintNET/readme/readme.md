# Как писать пояснительную записку (ПЗ)

Ориентир по стилю и структуре: файл `ПЗ_Голомидов.docx` в этой папке. Ниже — выжимка правил для текста проекта **«ЧТОТиБ Печать»** (`ChtotibDocsPrintNET`). При необходимости правьте формулировки под свою тему диплома и название организации.

## Структура документа (как в образце)

1. **Содержание** — нумерованные разделы с номерами страниц.
2. **Введение** — актуальность, проблема, тема и цель диплома, перечень задач, ожидаемый результат.
3. **1. Алгоритмизация**
   - 1.1. Постановка задачи  
   - 1.2. Описание задачи (подпункты 1.2.1, 1.2.2 …: общие сведения, назначение, функциональные и нефункциональные требования)  
   - 1.3. Хранение данных  
   - 1.4. Используемый язык и программное обеспечение  
   - 1.5. Используемые библиотеки  
   - 1.6. Системный анализ структуры базы данных  
   - 1.7. Архитектура проекта  
4. **2. Программирование** — по модулям/экранам (2.1, 2.2, 2.3.x …).
5. **3. Тестирование**
6. **Заключение**
7. **Список использованных источников**
8. **Приложение А** (листинги, скриншоты, схемы)

Текст ПЗ для этого репозитория ведём **пораздельно** в каталоге `readme/docs/` (файлы `01_Введение.md`, `02_…` и т.д.). Готовые фрагменты переносятся в Word вручную или через конвертацию.

## Стиль изложения (по образцу Голомидова)

| Приём | Как делать |
|--------|------------|
| Тон | Официально-учебный, от первого лица допустимо («цель **моего** проекта», «тема **моего** диплома») или безлично («целью проекта является…»). В одном документе — один вариант. |
| Абзацы | Развёрнутые (4–8 предложений). Сначала общий контекст (цифровизация, автоматизация), затем узкая предметная область. |
| Связки | «В наше время», «потому», «соответственно», «исходя из», «в связи с этим», «более того», «как следствие». |
| Термины | При первом упоминании — расшифровка: ИС, СУБД, PDF, QR-код, подложка бланка и т.п. |
| Перечни задач | После фразы «были сформированы следующие задачи:» — пункты **с новой строки**, в образце без маркеров; каждый пункт с **точкой с запятой** в конце, последний — с точкой. |
| Нумерация требований | В разделе «Описание задачи»: `1.2.1. Общие сведения`, `1.2.2. Назначение`, `1.2.3. Функциональные требования` с нумерованными подпунктами `1.`, `2.`, `3.` |
| Определения | Для сущностей предметной области: «X считается …», «X – объект, …» |
| Технические разделы | Конкретика: имена таблиц, технологий (.NET, WPF, SQL Server), модулей приложения — без лишней маркетинговой лексики. |
| Иллюстрации | В тексте ПЗ — отсылки «на рисунке X», «в таблице Y»; скриншоты — в приложение. |

## Чего избегать

- Коротких телеграфных абзацев из одного-двух предложений подряд во введении.
- Разговорных формулировок («крутое приложение», «просто нажать кнопку»).
- Копирования кода в основной текст (код — в приложение или в раздел «Программирование» фрагментами с пояснением).
- Противоречий между введением и разделом 1.2 (цель и задачи должны совпадать).
- **избегай длинное тире (—), используй только (–)!!!**
- НЕ НУЖНО ВЫДЕЛЕНИЙ ЖИРНЫМ ШРИФТОМ. Это диплом, а не реферат.

## Содержание введения (чеклист)

1. Актуальность IT и автоматизации в образовании / на предприятии.  
2. Описание **проблемы** вручную (печать дипломов и приложений, ошибки в расположении текста, время).  
3. **Тема диплома** "Создание прикладного ПО для печати документов об образовании".  
4. **Цель проекта** — одно ёмкое предложение.  
5. **Задачи** (7–10 пунктов) — от анализа до тестирования.  
6. Фраза про выбор **средств разработки** (язык, СУБД, библиотеки).  
7. **Ожидаемый результат** после внедрения.

## Именование системы в тексте

- Полное наименование: **«ЧТОТиБ Печать»** (настольное приложение для печати документов об образовании).  
- Кратко: **ИС**, **приложение**, **система** — после первого определения.
