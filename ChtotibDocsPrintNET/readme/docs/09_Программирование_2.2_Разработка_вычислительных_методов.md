# 2. Программирование (продолжение)

## 2.2. Разработка вычислительных методов

Под вычислительными методами в данной пояснительной записке понимаются алгоритмы прикладной логики, реализованные на языке C# в проекте «ЧТОТиБ Печать»: обращение к СУБД, проверка условий предметной области, подготовка данных для печати. Они сосредоточены в слое данных (каталог Data), сервисах (каталог Services) и командах ViewModel. Отдельные методы не дублируются на каждом экране: один и тот же код вызывается из карточки студента, из окна печати и при формировании PDF, что упрощает сопровождение системы после выпускной кампании. Ниже приведены фрагменты программы с пояснением назначения каждого блока; после кода – разбор ключевых строк и связь с работой пользователя. Места для скриншотов отмечены там, где уместно показать экран приложения. Структура базы данных и сценарии работы с таблицами – в п. 2.3.

### Пример 1. Инициализация доступа к базе данных и актуализация схемы

Перед началом работы приложение должно получить строку подключения к SQL Server и убедиться, что в уже развёрнутой базе есть все колонки, которые ожидает текущая версия программы. На практике база на сервере техникума часто создаётся один раз скриптом администратора, а программа на рабочих местах обновляется чаще; поэтому важно, чтобы новая сборка не требовала ручного ALTER TABLE на каждом компьютере. Для этого используется единый класс доступа к данным с шаблоном «одиночка» (singleton): при первом обращении создаётся один экземпляр, все экраны вызывают его через свойство Instance, и все операции чтения и записи идут через одну и ту же строку подключения.

В коде 2.1 представлен конструктор класса и метод EnsureSchemaPatches. Строка подключения читается из файла настроек пользователя; если файл пуст, подставляется вариант для LocalDB. Метод EnsureSchemaPatches выполняет идемпотентный SQL-скрипт: для каждой недостающей колонки в таблицах Students и Diplomas выполняется ALTER TABLE. Ошибки при патче перехватываются, чтобы запуск не прерывался на старых копиях базы с нестандартными правами.

Код 2.1 (инициализация DatabaseService и проверка схемы БД) представлен ниже.

```csharp
public static DatabaseService Instance => _instance ??= new DatabaseService();

private DatabaseService()
{
    _instance = this;
    var saved = AppSettings.LoadConnectionString();
    _connectionString = !string.IsNullOrEmpty(saved)
        ? saved
        : @"Server=(localdb)\MSSQLLocalDB;Database=ChtotibDocPrint;...";
    EnsureSchemaPatches();
}

private void EnsureSchemaPatches()
{
    using var conn = GetConnection();
    conn.Open();
    using var cmd = new SqlCommand(
        @"IF NOT EXISTS (SELECT 1 FROM sys.columns
              WHERE object_id = OBJECT_ID(N'dbo.Students') AND name = N'Qualification')
          ALTER TABLE dbo.Students ADD Qualification NVARCHAR(200) NULL;
          IF NOT EXISTS (... IssueDate ...)
          ALTER TABLE dbo.Diplomas ADD IssueDate DATE NULL;", conn);
    cmd.ExecuteNonQuery();
}
```

Свойство Instance с оператором ??= гарантирует ленивое создание объекта: память и соединение не расходуются до первого реального обращения к базе. В конструкторе сразу вызывается EnsureSchemaPatches, поэтому любой последующий запрос к студентам или дипломам уже видит поля Qualification и IssueDate. Условие IF NOT EXISTS в T-SQL исключает повторное добавление колонки при каждом запуске. Перехват исключений вокруг патча сделан намеренно мягким: если у учётной записи Windows нет прав на изменение схемы, программа всё равно стартует, а администратор увидит ошибку уже при сохранении данных в новые поля, а не при запуске exe-файла.

В результате при обновлении версии ИС на рабочих местах техникума не требуется повторно разворачивать базу: достаточно установить новую сборку exe. Все экраны, обращающиеся к DatabaseService.Instance, автоматически получают актуальную схему и единую строку подключения, что исключает ситуацию, когда в одном разделе данные читаются из одной базы, а в другом – из устаревшей копии настроек.

### Пример 2. Проверка подключения к серверу базы данных

Пользователь вводит строку подключения на странице «Настройки» и должен сразу узнать, доступен ли сервер. Это типичная ситуация при первой установке ИС в учебной части: сотрудник копирует строку от администратора, вставляет в поле и нажимает проверку, не перезапуская приложение и не открывая SSMS. Вычислительный метод TestConnection открывает соединение SqlConnection в блоке try, не сохраняя пароль в отдельных полях – используется готовая строка из интерфейса. При успехе в статусную строку записываются DataSource и имя базы; при исключении – текст ошибки (неверный логин, нет сети, база не найдена). Сохранение строки в файл настроек выполняется отдельной командой SaveAll, чтобы случайная проверка с опечаткой не перезаписала рабочую конфигурацию.

Код 2.2 (метод проверки подключения, SettingsViewModel) представлен ниже.

```csharp
[RelayCommand]
private void TestConnection()
{
    try
    {
        using var conn = new SqlConnection(ConnectionString);
        conn.Open();
        StatusMessage = $"✓ Подключение успешно! Сервер: {conn.DataSource}, БД: {conn.Database}";
        IsError = false;
        HasStatus = true;
    }
    catch (Exception ex)
    {
        StatusMessage = $"✗ Ошибка подключения: {ex.Message}";
        IsError = true;
        HasStatus = true;
    }
}
```

Конструкция using обеспечивает закрытие соединения даже при сбое: после проверки на сервере не остаётся «висящих» сессий от каждого нажатия кнопки. Атрибут RelayCommand связывает метод с кнопкой «Проверить подключение» без обработчика Click в XAML, что соответствует шаблону MVVM, принятому в проекте. Флаги IsError и HasStatus управляют цветом панели статуса на форме: пользователь визуально отличает удачную проверку от ошибки, не читая префикс символа в тексте сообщения. Сообщение с именем сервера и базы помогает убедиться, что подключение идёт именно к ChtotibDocPrint на нужном экземпляре SQL, а не к другой базе на том же компьютере.

После успешной проверки сотрудник нажимает «Сохранить всё»: строка записывается в settings.json, а DatabaseService.SetConnectionString применяет её ко всем последующим запросам в текущем сеансе. Таким образом, проверка и сохранение разделены: ошибочная строка не попадёт в файл настроек, если пользователь только проверил соединение и не подтвердил сохранение.

### Пример 3. Выборка оценок выпускника с учётом специальности группы

Для приложения к диплому в таблицу попадают не все предметы из справочника, а только относящиеся к специальности учебной группы выпускника и дисциплины, помеченные как общие (SpecialtyId IS NULL). В техникуме один и тот же справочник предметов используется для нескольких направлений; без фильтра в приложении оказались бы, например, дисциплины смежной специальности, что недопустимо для официального бланка. Метод GetStudentGrades формирует SQL-запрос с соединением Grades, Subjects, Students и Groups и фильтром по идентификатору студента, чтобы правило отбора выполнялось на стороне сервера, а не в цикле на клиенте.

Код 2.3 (запрос оценок с фильтром по специальности) представлен ниже.

```csharp
public List<Grade> GetStudentGrades(int studentId)
{
    var list = new List<Grade>();
    using var conn = GetConnection();
    conn.Open();
    using var cmd = new SqlCommand(
        @"SELECT g.Id, g.StudentId, g.SubjectId, g.Grade, g.GradeType,
                 sub.Name, sub.Course, sub.Hours
          FROM Grades g
          INNER JOIN Subjects sub ON g.SubjectId = sub.Id
          INNER JOIN Students st ON st.Id = g.StudentId
          INNER JOIN Groups grp ON grp.Id = st.GroupId
          WHERE g.StudentId = @sid
            AND (sub.SpecialtyId IS NULL OR sub.SpecialtyId = grp.SpecialtyId)
          ORDER BY sub.Course, sub.Name", conn);
    cmd.Parameters.AddWithValue("@sid", studentId);
    using var r = cmd.ExecuteReader();
    while (r.Read())
        list.Add(new Grade { /* поля из reader */ });
    return list;
}
```

Параметр @sid передаётся отдельно от текста запроса, что исключает подстановку посторонних символов извне и соответствует требованиям безопасности при работе с персональными данными. Условие в скобках реализует правило учебной части: общие предметы видны всем специальностям, остальные – только своей. Сортировка ORDER BY задаёт порядок строк в печатной таблице по курсу и названию, как принято в бумажном приложении к диплому. Результат метода – список объектов Grade с уже подставленными именами предметов и часами, поэтому экран печати и модуль PDF не обращаются к справочнику Subjects повторно.

Тот же метод используется при расчёте признака «диплом с отличием»: StudentHonorEvaluator получает именно этот набор оценок и проверяет итоговые баллы. Согласованность данных между приложением к диплому, предпросмотром и логикой отличия обеспечивается одним источником на стороне SQL Server.

### Пример 4. Сохранение реквизитов диплома (алгоритм «обновить или вставить»)

В карточке студента серия и номер бланка хранятся в таблице Diplomas, а регистрационный номер и квалификация – в Students. Такое разделение отражает предметную область: реквизиты государственного образца (серия, номер, дата выдачи) относятся к экземпляру бланка, а регистрационный номер для приложения – к личному делу выпускника. Метод UpdateStudentAndDiplomaBlank выполняет два шага в одном открытом соединении: UPDATE по студенту, затем UpsertDiplomaBlank, чтобы сотрудник одной кнопкой «Сохранить» не оставлял половину данных в базе. Вспомогательный метод сначала считает записи COUNT(1) по StudentId; если запись есть – UPDATE, иначе – INSERT с признаком IsPrinted = 0 (документ ещё не выводился на принтер).

Код 2.4 (обновление студента и upsert диплома) представлен ниже.

```csharp
public void UpdateStudentAndDiplomaBlank(int studentId, string? registrationNumber, ...)
{
    using var conn = GetConnection();
    conn.Open();
    using (var cmd = new SqlCommand(
        @"UPDATE Students SET RegistrationNumber=@rn, Qualification=@q, ... WHERE Id=@id", conn))
    {
        cmd.Parameters.AddWithValue("@id", studentId);
        cmd.Parameters.AddWithValue("@rn",
            string.IsNullOrWhiteSpace(registrationNumber) ? DBNull.Value : registrationNumber);
        cmd.ExecuteNonQuery();
    }
    UpsertDiplomaBlank(conn, studentId, diplomaSeries, diplomaNumber, diplomaType, diplomaIssueDate);
}

private static void UpsertDiplomaBlank(SqlConnection conn, int studentId, ...)
{
    var exists = Convert.ToInt32(
        new SqlCommand("SELECT COUNT(1) FROM Diplomas WHERE StudentId=@sid", conn)
            { Parameters = { new("@sid", studentId) } }.ExecuteScalar()) > 0;
    if (exists)
        /* UPDATE Diplomas SET Series, Number, DiplomaType, IssueDate */;
    else
        /* INSERT INTO Diplomas (...) VALUES (...), IsPrinted = 0 */;
}
```

Пустые строки в параметрах заменяются на DBNull.Value, чтобы в базе хранился NULL, а не пробелы: отчёты и предупреждение на главной странице о незаполненных полях работают корректно. Алгоритм upsert гарантирует связь «один студент – одна запись о бланке» без дублирования при повторном сохранении карточки; иначе при каждом исправлении серии создавалась бы вторая строка в Diplomas с тем же StudentId. Поле DiplomaType в той же операции сохраняет тип «Обычный» или «С отличием», который затем читает модуль печати при выборе цвета подложки.

Использование одного открытого соединения для UPDATE Students и upsert Diplomas снижает риск рассинхронизации: если второй шаг завершится с ошибкой, администратор увидит это в сообщении, а не получит студента с обновлённым регистрационным номером, но без записи о бланке. При повторном открытии карточки сотрудник видит согласованный комплект полей, готовый к печати.

### Пример 5. Определение квалификации и типа диплома с отличием

Для оборотной стороны диплома нужна строка квалификации; для выбора цвета бланка – признак «с отличием». Оба признака зависят не только от введённых вручную значений, но и от справочника специальностей и таблицы оценок, поэтому их нельзя зашивать в XAML как константы. Эти правила вынесены в отдельные статические классы, чтобы не дублировать их в предпросмотре, при сохранении PDF и при смене студента в очереди печати: любое изменение порядка проверки делается в одном месте.

Класс QualificationResolver последовательно проверяет: значение в карточке студента; поле Qualification в специальности; текст в скобках в конце названия специальности (регулярное выражение); код специальности по умолчанию (например, 09.02.07 – «Программист»).

Класс StudentHonorEvaluator возвращает true, если в базе указан тип «С отличием» или все итоговые оценки (кроме курсовых) равны 5.

Код 2.5 (определение квалификации и диплома с отличием) представлен ниже.

```csharp
// QualificationResolver.cs
public static string Resolve(string? studentQualification, Specialty? specialty)
{
    if (!string.IsNullOrWhiteSpace(studentQualification))
        return studentQualification.Trim();
    if (specialty != null && !string.IsNullOrWhiteSpace(specialty.Qualification))
        return specialty.Qualification.Trim();
    var fromName = TryExtractFromSpecialtyName(specialty?.Name);
    if (!string.IsNullOrEmpty(fromName)) return fromName;
    return specialty?.Code switch
    {
        "09.02.07" => "Программист",
        "09.02.06" => "Специалист по защите информации",
        _ => "",
    };
}

// StudentHonorEvaluator.cs
public static bool ShouldUseHonorTemplate(string? diplomaType, IEnumerable<Grade>? grades) =>
    IsHonorDiplomaType(diplomaType) || AllFinalGradesAreExcellent(grades);

public static bool AllFinalGradesAreExcellent(IEnumerable<Grade>? grades)
{
    var final = grades?.Where(g => g.GradeType != "Курсовая").ToList();
    return final is { Count: > 0 } && final.All(g => g.GradeValue == 5);
}
```

Выражение switch по коду специальности закрывает случаи, когда в справочнике не заполнено поле Qualification, но код ФГОС известен (типично для демонстрационных и импортированных данных). Регулярное выражение для скобок в названии специальности позволяет использовать формулировки вида «… (Программист)» без отдельного поля в базе. Метод ShouldUseHonorTemplate объединяет два основания из ТЗ: явный тип диплома и академическое отличие по оценкам; достаточно выполнения одного из условий. Курсовые работы в расчёт не входят, так как на бланке они выводятся отдельным блоком и не считаются итоговой аттестацией по предмету в смысле приложения. Если список итоговых оценок пуст, признак отличия не выставляется автоматически, чтобы не напечатать красный бланк выпускнику без внесённых оценок.

При смене студента в окне «Печать документов» эти методы вызываются снова: квалификация на обороте и цвет подложки обновляются без перезапуска приложения. Это важно при пакетной проверке выпускников одной группы, когда сотрудник листает очередь и сверяет, что у отличников действительно подставился красный бланк, а у остальных – синий.

### Пример 6. Формирование полезной нагрузки для QR-кода диплома

Одна из задач ИС – облегчить проверку реквизитов диплома и снизить риск «ручной перепечатки» серии, номера и кода демонстрационного экзамена. Для этого на оборотной стороне диплома печатается QR-код, содержащий компактную строку. Выбор именно текстовой полезной нагрузки удобен тем, что любой стандартный сканер на телефоне сможет показать содержимое как текст, а при необходимости его можно обработать программно.

В коде 2.6 представлен метод Build, формирующий JSON-строку из ФИО выпускника, кода демо-экзамена и реквизитов бланка. Поля дополнительно очищаются от лишних пробелов. Для корректной передачи русских символов используется настройка сериализации с отключением избыточного экранирования, чтобы строка оставалась читаемой и не раздувалась по длине.

Код 2.6 (формирование строки для QR-кода) представлен ниже.

```csharp
public static string Build(Student student, string? demoExamCode, string? diplomaSeries, string? diplomaNumber)
{
    var payload = new
    {
        fio = student.FullName?.Trim() ?? "",
        demo = (demoExamCode ?? "").Trim(),
        series = (diplomaSeries ?? "").Trim(),
        number = (diplomaNumber ?? "").Trim(),
    };
    return JsonSerializer.Serialize(payload, new JsonSerializerOptions
    {
        Encoder = JavaScriptEncoder.UnsafeRelaxedJsonEscaping,
        WriteIndented = false,
    });
}
```

Поля fio, demo, series и number выбраны как краткие ключи JSON, чтобы уменьшить длину строки в QR: чем короче полезная нагрузка, тем крупнее модули кода на бланке и проще считывание камерой. Код демонстрационного экзамена подставляется из кода специальности группы выпускника (например, 09.02.07), что согласовано с требованиями к содержимому QR в демонстрационной версии системы.

Таким образом, QR-код переносит на бумажный документ те же ключевые реквизиты, которые сотрудник видит в карточке студента. Это повышает удобство проверки и снижает вероятность ошибок при повторном вводе данных из диплома в сторонних системах. При сканировании на смартфоне отображается читаемый JSON-текст, который при необходимости можно сверить с данными в ИС.

### Пример 7. Генерация PNG изображения QR-кода

После получения полезной нагрузки требуется превратить её в графическое изображение, пригодное для вставки в предпросмотр и PDF. В ИС используется генерация QR в формате PNG. Такой формат прост для встраивания в графический конвейер, а чёрно-белая палитра даёт предсказуемый результат при печати на большинстве устройств.

В коде 2.7 представлен метод RenderPng. Он задаёт уровень коррекции ошибок L, который даёт меньшую матрицу при той же длине строки и, как следствие, более крупные модули – это важный практический момент для сканирования камерой с бумаги. В методе предусмотрена защитная логика: если исходная строка слишком длинная для генератора, она ограничивается по длине, чтобы печать не падала с исключением.

Код 2.7 (рендер QR в PNG) представлен ниже.

```csharp
private const int PixelsPerModule = 12;

public static byte[]? RenderPng(string text)
{
    if (string.IsNullOrWhiteSpace(text)) return null;
    try
    {
        using var gen = new QRCodeGenerator();
        using var data = gen.CreateQrCode(text, QRCodeGenerator.ECCLevel.L);
        var png = new PngByteQRCode(data);
        return png.GetGraphic(PixelsPerModule, new byte[] { 0, 0, 0, 255 }, new byte[] { 255, 255, 255, 255 }, true);
    }
    catch
    {
        return null;
    }
}
```

Константа PixelsPerModule задаёт плотность растра: при масштабировании в PDF изображение не «размывается» до нечитаемого состояния. Уровень коррекции L выбран осознанно: для короткой JSON-строки матрица QR остаётся компактной, а модули – крупными. Блок try/catch с возвратом null не прерывает печать всего комплекта: при сбое генерации на бланке просто не будет QR, но ФИО и подписи сохранятся.

Полученное изображение далее вставляется в рассчитанную область на бланке диплома. Благодаря единым настройкам ECC и размера модуля QR получается стабильным по качеству при предпросмотре и печати. Одинаковый алгоритм используется и в WPF-предпросмотре, и при экспорте в PDF, поэтому сотрудник видит на экране то же, что получит на принтере.

### Пример 8. Вставка QR-кода в PDF с учётом масштабирования

При формировании PDF координаты текстовых блоков и изображений пересчитываются из «координат предпросмотра» в реальные координаты страницы. Особенность QR-кода заключается в том, что он должен оставаться квадратным: если растянуть его по ширине и высоте по-разному, модули станут прямоугольными и сканирование ухудшится. Поэтому при отрисовке QR выбирается единый коэффициент масштаба по меньшей стороне.

В коде 2.8 представлен метод DrawDiplomaBackQr: он проверяет, что QR включён (размер больше минимального), получает PNG байты и вычисляет прямоугольник назначения. Далее QR центрируется внутри ячейки, чтобы при различии sx и sy не создавалось смещения «в угол» и визуально сохранялось аккуратное расположение в рамке.

Код 2.8 (отрисовка QR-кода на обороте диплома) представлен ниже.

```csharp
if (layout.QrSize < 4) return;
var text = diplomaQrPayload?.Trim() ?? "";
if (text.Length == 0) return;
var pngBytes = DiplomaQrCodeService.RenderPng(text);
if (pngBytes == null || pngBytes.Length == 0) return;

using var qrBmp = SKBitmap.Decode(pngBytes);
var cellW = (float)(layout.QrSize * sx);
var cellH = (float)(layout.QrSize * sy);
var s = Math.Min(sx, sy);
var side = (float)(layout.QrSize * s);
var left = (float)Math.Round(cellLeft + (cellW - side) * 0.5);
var top = (float)Math.Round(cellTop + (cellH - side) * 0.5);
canvas.DrawBitmap(qrBmp, new SKRect(0, 0, qrBmp.Width, qrBmp.Height), dest, paint);
```

Переменные sx и sy отражают разный масштаб предпросмотра по осям: область под QR в макете может быть чуть вытянутой, но сам код остаётся квадратным за счёт Math.Min(sx, sy). Округление координат left и top уменьшает субпиксельное «дрожание» границ при печати. Если полезная нагрузка пуста или PNG не сгенерировался, метод завершается без ошибки – остальные поля оборота диплома печатаются как обычно.

За счёт такого расчёта QR сохраняет правильную геометрию и одинаково хорошо читается как в PDF, так и на бумаге. Это особенно важно для массовой печати, когда документ сканируется «на ходу» и нет времени подбирать угол и фокус камеры. Координаты QrLeft, QrTop и QrSize берутся из сохранённого профиля раскладки, поэтому смещение QR на бланке настраивается в интерфейсе печати и сохраняется между сеансами.

### Пример 9. Выбор шаблона бланка и поддержка пользовательских подложек

ИС позволяет работать как со встроенными JPG-шаблонами бланков, так и с пользовательскими файлами, если в техникуме применяются другие отсканированные подложки. Для этого реализован сервис, который хранит пути к шаблонам и по типу документа возвращает байты изображения. Для диплома предусмотрены варианты «обычный» и «с отличием», так как у них разные подложки.

В коде 2.9 представлен метод LoadTemplateBytes. Он сначала пытается прочитать пользовательский путь из файла настроек шаблонов; если путь не задан или файл отсутствует, берётся встроенный вариант из каталога DocsTemplates. Такое поведение позволяет администратору один раз заменить шаблон, после чего все печати идут с нужной подложкой без ручного выбора файла перед каждой операцией.

Код 2.9 (загрузка подложки бланка JPG по типу документа) представлен ниже.

```csharp
public static byte[]? LoadTemplateBytes(string documentType, bool useHonor = false)
{
    var settings = LoadSettings();
    var fileName = ResolveFileName(documentType, useHonor);
    if (fileName == null) return null;

    var customPath = settings.GetPath(documentType, useHonor);
    if (!string.IsNullOrWhiteSpace(customPath) && File.Exists(customPath))
        return File.ReadAllBytes(customPath);

    var bundled = Path.Combine(TemplatesDirectory, fileName);
    if (!File.Exists(bundled)) return null;
    return File.ReadAllBytes(bundled);
}
```

Параметр useHonor переключает имена файлов между обычным дипломом и вариантом «с отличием» (DiplomaExcellent_Front.jpg и аналоги). Сначала проверяется пользовательский путь из template_settings.json – это позволяет подключить отсканированные бланки конкретного техникума; если файла нет, подставляется встроенный шаблон из папки DocsTemplates рядом с exe. Возврат null сигнализирует модулю печати, что подложку показать нельзя (режим «только текст» остаётся рабочим).

Таким образом, сервис шаблонов отделяет «где лежит подложка» от логики печати и отрисовки. При изменении комплекта бланков достаточно обновить настройки и файлы шаблонов, не затрагивая алгоритмы формирования PDF. На экране печати сотрудник сразу видит смену подложки при переключении типа диплома или студента с отличием.

### Пример 10. Разбор строк импорта предметов с проверками формата

При подготовке к выпускной кампании часто требуется быстро загрузить дисциплины из внешнего источника. Чтобы снизить число ошибок при импорте, приложение поддерживает несколько вариантов формата строки: предмет может быть общим для всех специальностей, привязанным по коду или названию специальности, либо загруженным по «устаревшему» формату, где во второй колонке указан курс.

### Пример 13. Отображение статистики готовности данных на главной странице

Для удобства работы в период выпускной кампании на главной странице выводятся агрегированные показатели: сколько студентов находится в базе, сколько выпускников, процент заполненности регистрационных номеров у выпускников выпускающих групп, сколько документов напечатано сегодня, а также дата последней печати. Расчёт выполняется в модели представления главной страницы: она обращается к слою данных, получает целые значения и формирует строки, которые затем отображаются на карточках статистики. Код 2.13 (загрузка статистики для главной страницы) представлен ниже.

```csharp

```

GetTotalStudents и GetTotalGraduates дают общую картину наполнения базы. GetDataCompletionStats считает только выпускников выпускающих групп с заполненным RegistrationNumber – это напрямую связано с предупреждением на той же странице (HasWarning). Показатели обновляются при каждом открытии раздела «Главная»; при ошибке подключения блок остаётся пустым, но приложение продолжает работу – сотрудник может перейти в «Настройки» и исправить строку подключения.

### Пример 14. Добавление оценки через диалог и защита от дублирования предмета

Оценки для приложения к диплому вводятся в карточке студента. Чтобы сотрудник учебной части не ошибся и не добавил два раза один и тот же предмет, в ИС реализована защита от дублирования. Она работает в два этапа. Во-первых, в диалоге добавления оценки пользователю показываются только те дисциплины, по которым у выбранного студента ещё нет записи. Во-вторых, при сохранении выполняется дополнительная проверка в базе – если оценка по предмету уже существует, вставка блокируется и пользователь получает понятное сообщение.

Код 2.14 (загрузка списка предметов без оценок и сохранение) представлен ниже.

```csharp
// AddGradeDialog.xaml.cs
var subjects = db.GetSubjectsAvailableForStudentGrade(st.GroupId, _studentId);
CbSubject.ItemsSource = subjects;
if (subjects.Count == 0)
{
    ShowNoSubjectsState("Нет предметов без оценки: по всем доступным предметам оценка уже выставлена.");
    return;
}

// при нажатии «Добавить»
DatabaseService.Instance.InsertGrade(_studentId, subject.Id, gradeValue, gradeType);
```

Метод GetSubjectsAvailableForStudentGrade формирует список доступных дисциплин с учётом специальности группы и исключает предметы, по которым уже есть запись в Grades. Благодаря этому в выпадающем списке нет «лишних» вариантов, и пользователь не тратит время на поиск нужной строки среди уже заполненных предметов. Если список пуст, диалог показывает красное сообщение и блокирует кнопку добавления – это предотвращает ситуацию, когда сотрудник нажимает «Добавить», не понимая, почему ничего не происходит.

Дополнительная проверка в InsertGrade защищает от редких случаев параллельного ввода (например, когда карточка студента открыта на двух компьютерах). Даже если дисциплина каким-то образом снова попадёт в список, база не примет повторную запись, и окно покажет предупреждение.

### Пример 15. Предотвращение удаления специальности при наличии связанных данных

В справочнике специальностей хранится базовая информация, к которой привязаны группы и предметы. Если удалить специальность, к которой уже относятся учебные группы, база данных потеряет смысл связей, а у выпускников исчезнет контекст для печати (код, квалификация и др.). Поэтому в интерфейсе предусмотрено подтверждение удаления, а на уровне базы удаление блокируется внешними ключами. В результате пользователь получает понятное сообщение и не может случайно разрушить структуру данных.

Код 2.15 (команда удаления специальности и обработка ошибки) представлен ниже.

```csharp
// AdminViewModel.cs
if (MessageBox.Show($"Удалить специальность «{SelectedSpecialty.Name}»?\n\nЕсли к ней привязаны группы/предметы, удаление не получится.",
        "Удаление", MessageBoxButton.YesNo, MessageBoxImage.Warning) != MessageBoxResult.Yes)
    return;

try
{
    DatabaseService.Instance.DeleteSpecialty(SelectedSpecialty.Id);
    LoadAll();
}
catch (Exception ex)
{
    MessageBox.Show($"Не удалось удалить специальность: {ex.Message}", "Специальности",
        MessageBoxButton.OK, MessageBoxImage.Warning);
}
```

Первое окно предупреждает пользователя о возможных последствиях и даёт шанс отменить действие. Если пользователь подтверждает удаление, команда обращается к слою данных. Если специальность используется в группах или предметах, SQL Server возвращает ошибку ограничения целостности, которая отображается в виде предупреждения. Такой подход удобен для учебной части: сотрудник видит причину отказа и понимает, что сначала нужно перенести группы или удалить зависимые записи, а уже потом менять справочник.

На рисунке 2.33 показано предупреждение перед удалением специальности, а на рисунке 2.34 – сообщение об ошибке при попытке удалить специальность, к которой привязаны группы.
