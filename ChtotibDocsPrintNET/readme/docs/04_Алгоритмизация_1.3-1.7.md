## 1.3. Хранение данных

Все учётные и печатные сведения информационной системы «ЧТОТиБ Печать» хранятся в локальной базе данных на платформе Microsoft SQL Server (имя базы – ChtotibDocPrint). Под «локальной» понимается размещение экземпляра СУБД на сервере или рабочей станции в сети техникума (например, SQL Server Express или LocalDB на компьютере учебной части), без обязательного выноса данных в облако. Такой вариант соответствует политике многих образовательных организаций: персональные данные выпускников и реквизиты документов остаются под контролем учреждения.

Клиентское приложение не содержит собственного «встроенного» хранилища списков студентов и оценок: оно обращается к серверу базы данных по строке подключения. Строка подключения задаёт администратор (имя сервера, имя базы, способ аутентификации) и указывает, к какому экземпляру SQL Server и какой базе нужно подключиться. После установки параметров приложение использует эту строку при каждом запросе к данным: выборка групп и студентов, сохранение оценок, реквизитов диплома, записей в журнал печати и т.д. При недоступности сервера операции с данными не выполняются, что отделяет работу с документами от риска рассинхронизации копий на разных компьютерах.

В качестве модели данных выбрана реляционная модель. Сведения разбиты на связанные таблицы (специальности, группы, студенты, предметы, оценки, дипломы, члены комиссии, история печати) и связаны отношениями «один ко многим» и «один к одному» через ключи. Например, один студент принадлежит одной группе, группа относится к одной специальности; у выпускника может быть одна запись о дипломе и множество оценок по разным дисциплинам. Представления (view) используются для типовых контрольных выборок, например списка выпускников без регистрационного номера.

Данный способ хранения наиболее подходит для поставленной задачи по следующим причинам.

Во-первых, предметная область изначально нормализована: одни и те же специальности и предметы используются для многих групп и студентов, а оценки и реквизиты диплома должны однозначно относиться к конкретному выпускнику. Реляционная СУБД исключает дублирование справочников в «плоских» таблицах Excel и снижает риск противоречивых данных.

Во-вторых, требуется целостность: нельзя удалить специальность, если к ней привязаны группы, нельзя завести оценку без студента и предмета. Ограничения и внешние ключи в SQL Server поддерживают эти правила на уровне базы, а не только в интерфейсе программы.

В-третьих, в период выпускной кампании важны одновременная работа нескольких операций (поиск студента, массовая печать, импорт из учётной системы) и сохранность данных при сбое. Транзакционная модель SQL Server обеспечивает согласованную запись пакета изменений; резервное копирование и восстановление базы входят в стандартную практику администрирования техникума.

В-четвёртых, техникум уже может использовать Microsoft SQL Server или совместимый экземпляр для других задач; развёртывание отдельной базы ChtotibDocPrint не требует смены платформы и позволяет при необходимости обмениваться данными с 1С через импорт, не подменяя учётную систему, а накапливая в ИС актуальный срез для печати.

Импорт из табличных файлов и учётных систем рассматривается как способ наполнения базы, а не как основное хранилище: после загрузки все операции печати и отчёты опираются на единый центр данных в СУБД.

C# (Си Шарп) – основной язык прикладной логики: работа с базой данных, импорт сведений, формирование PDF, печать, обработка команд интерфейса. Разметка интерфейса отделена от кода на C#, что упрощает вёрстку экранов «Группы», «Студенты», «Печать документов» и настройку тем оформления. XAML выбран потому, что он является штатным языком разметки WPF и поддерживает привязку к свойствам ViewModel без ручного обновления каждого элемента на экране.

SQL (в диалекте Transact-SQL, T-SQL) – язык описания и обслуживания базы данных ChtotibDocPrint: создание таблиц, ключей, представлений, выборки и изменения данных при выполнении запросов из приложения. SQL применяется там, где нужна централизованная фильтрация и соединение таблиц (например, студенты выпускающей группы с оценками и реквизитами диплома). Выбор обусловлен тем, что в качестве СУБД используется SQL Server; тот же язык применяют администраторы техникума в SSMS для резервного копирования и контроля данных.

SSMS не входит в состав приложения «ЧТОТиБ Печать», но необходима на этапе внедрения и сопровождения; выбор обусловлен тем, что это стандартная бесплатная утилита к SQL Server.

Операционная система Microsoft Windows (версии 10 и выше, 64-разрядная) – среда, в которой работают WPF-клиент, драйвер принтера и экземпляр SQL Server. Windows выбрана как целевая ОС рабочих мест учебной части техникума; разработка под одну платформу упрощает тестирование печати и предпросмотра документов.

Дополнительно на компьютере пользователя используется драйвер установленного принтера (средство ОС Windows) для вывода готовых PDF на печать; без него функция печати комплекта документов недоступна, хотя формирование и сохранение PDF в ИС сохраняется.

## 1.6. Системный анализ структуры базы данных

В проекте используется реляционная база данных ChtotibDocPrint: сведения представлены в виде таблиц, отражающих сущности предметной области, и полей, отражающих атрибуты этих сущностей. Во всех таблицах задан первичный ключ Id (целое число с автоувеличением), по которому записи однозначно идентифицируются и связываются внешними ключами. Такая организация соответствует правилам нормализации: справочная информация (специальности, предметы, состав комиссии) отделена от данных о конкретных выпускниках и результатах их обучения, что уменьшает дублирование и упрощает сопровождение базы в техникуме.

Исходной для учёта выпускников является цепочка «специальность – группа – студент»; к студенту привязываются оценки, реквизиты диплома и записи журнала печати. На рисунке 1 (в приложении к пояснительной записке) может быть приведена ER-диаграмма связей между таблицами. Ниже каждая сущность описана с указанием назначения, состава хранимых данных и порядка использования в информационной системе «ЧТОТиБ Печать».

Справочник направлений подготовки представлен сущностью Specialties (специальности). Таблица предназначена для единого хранения кодов и наименований специальностей среднего профессионального образования, сроков и формы обучения, присваиваемой квалификации. Без этой таблицы пришлось бы повторять одни и те же тексты в каждой группе и в печатных формах. При создании учебной группы в разделе «Группы» пользователь выбирает специальность из списка; при печати диплома на лицевую сторону подставляются наименование и квалификация, соответствующие записи в Specialties. Значение DiplomaType используется при выборе подложки бланка (в том числе вариант «с отличием») и при выводе надписи «Дубликат» в макете. Серия и номер выводятся на оборотной стороне диплома и включаются в содержимое QR-кода. Поле IsPrinted позволяет отметить, что документ уже проходил через печать. Заполнение таблицы выполняется в карточке студента или при импорте; без неё невозможно корректно сформировать оборот диплома. Поля сущности приведены в таблице 6.

Система использует это представление на главной странице (показатель «выпускники без регистрационного номера») и при подготовке к печати, чтобы сотрудник заранее исправил недостающие реквизиты в приложении к диплому.

Разработанная база данных представляет собой структурированную систему для централизованного хранения сведений о выпускниках и параметров, необходимых для печати диплома и приложения. Связанные таблицы обеспечивают целостность данных за счёт внешних ключей; удаление специальности или предмета при наличии зависимых записей ограничивается приложением. Процедура обслуживания базы удаляет ошибочные «сиротские» строки в Grades, Diplomas и PrintHistory после импорта или ручного удаления карточек, поддерживая согласованность данных на протяжении всей выпускной кампании.

## 1.7. Архитектура проекта

Проект ChtotibDocsPrintNET построен по многослойной схеме с разделением ответственности (MVVM – Model–View–ViewModel).

Слой Models содержит классы предметной области: Student, Group, Specialty, Subject, Grade, Diploma, CommissionMember, а также модели раскладки печати (DiplomaFrontOverlayLayout, DiplomaBackOverlayLayout, AppendixFrontOverlayLayout, AppendixBackOverlayLayout, PreviewCropInsets и др.).

Слой Data – класс DatabaseService: единая точка доступа к SQL Server (CRUD, отчёты для главной страницы, выборки для печати) по заданной строке подключения.

Слой Services – прикладная логика, не привязанная к конкретному экрану:

- ImportService, OneCImportService – загрузка данных из файлов и 1С;

- DocumentTemplateService – подложки бланков;

- DiplomaPrintService, DiplomaPdfSkiaCompositor, StudentDiplomaPrintHelper – подготовка данных и сборка PDF;

- DiplomaQrCodeService, DiplomaQrPayloadBuilder – QR-код;

- PdfPrintService – отправка PDF на принтер;

- DatabaseMaintenanceService, SchemaSlimmingService – обслуживание БД;

- OneCSettingsService, PersonNameFormatter – вспомогательные функции.

Слой ViewModels – MainViewModel и модели страниц (Dashboard, Groups, GroupDetail, Students, StudentDetail, Print, Admin, Settings). Навигация между экранами через MainViewModel.NavigateTo; команды – RelayCommand.

Слой Views – XAML-страницы (DashboardPage, GroupsPage, PrintPage, AdminPage, SettingsPage и диалоги добавления записей, мастер OneCConnectionWizard). Связь с ViewModel через DataContext и DataTemplate в MainWindow.

Каталоги проекта: Data, Models, Services, ViewModels, Views, Views\Dialogs, Converters, Database (SQL-скрипты), DocsTemplates (подложки), Resources (иконки, логотип), readme\docs (текст ПЗ).

Поток данных при печати: View (PrintPage) – PrintViewModel загружает студентов из базы данных и параметры макета – сервисы формирования PDF – вывод на принтер или сохранение файла. Такое разделение упрощает сопровождение: учётные данные сосредоточены в СУБД, а логика отображения и печати вынесена в отдельные модули приложения.
